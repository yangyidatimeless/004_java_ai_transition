## 📋 议题概述

随着 AI 技术的快速发展，传统的 Java 服务端开发工程师面临着巨大的转型压力。本次讨论旨在探讨：**资深 Java 服务端工程师如何在 AI 时代保持竞争力，需要学习哪些新技术，以及如何调整职业发展路径。**

---

## 🎯 核心结论

### 1. 转型策略：渐进式融合

**共识**：不抛弃 Java 技术栈，而是在现有基础上叠加 AI 能力。

```
转型路径：
第一阶段（已完成）：AI 辅助开发
├── 熟练使用 Cursor/Copilot
└── 建立与 AI 的协作工作流

第二阶段（进行中）：AI API 集成
├── 学习 LLM API 调用
├── 掌握 Prompt Engineering
├── 了解 RAG 架构
└── 实践：在现有项目中集成 AI 功能

第三阶段（规划中）：AI 原生应用开发
├── 向量数据库集成
├── Agent 框架学习（Spring AI/LangChain4j）
└── 实践：独立设计 AI 驱动的服务

第四阶段（长期）：AI 基础设施
├── 模型部署与推理优化
├── MLOps 基础理解
└── 成本与性能平衡策略
```

### 3. 技术标准

#### 3.1 API 规范（前后端对齐）

**流式响应格式（SSE）**：
```json
data: {"type": "start", "conversationId": "abc123"}
data: {"type": "thinking", "stage": "searching_knowledge", "message": "正在检索知识库..."}
data: {"type": "chunk", "content": "你好，", "contentType": "text", "tokenUsage": 10}
data: {"type": "chunk", "content": "https://example.com", "contentType": "link"}
data: {"type": "end", "totalTokens": 50, "latency": 1200}
data: {"type": "error", "code": "RATE_LIMITED", "message": "API 限流，请稍后重试"}
```

**错误码定义**：
| 错误码 | HTTP 状态码 | 说明 | 前端处理建议 |
|--------|------------|------|-------------|
| `LLM_TIMEOUT` | 504 | LLM API 超时 | 显示"AI 正在思考，请稍后" |
| `RATE_LIMITED` | 429 | 触发限流 | 显示"请求太多，稍后再试" |
| `INVALID_PROMPT` | 400 | Prompt 违规 | 显示"问题不合规，请重新提问" |
| `CONTEXT_TOO_LONG` | 413 | 上下文超出限制 | 提示"对话太长，建议开启新话题" |
| `SERVICE_UNAVAILABLE` | 503 | AI 服务不可用 | 降级到人工客服或普通搜索 |

**置信度计算**：
- 基于检索质量间接计算（检索文档数量 × 相似度）
- API 返回包含 `confidence` 和 `confidenceLevel` 字段
- 前端根据置信度显示不同提示（高置信度正常显示，低置信度建议人工确认）

#### 3.2 可测试性设计

**原则**：
- 可观测：记录所有 AI 调用的输入输出
- 可 Mock：LLM 调用抽象为接口，支持依赖注入
- 可回放：录制典型对话，用于回归测试
- 可追溯：每个回答记录来源文档
- 可度量：记录响应时间、token 消耗、置信度

### 4. 质量标准

#### 4.1 AI 代码审查清单

- [ ] AI 生成的代码必须有标注（`// Generated by AI, reviewed by XXX`）
- [ ] 必须有对应的单元测试（覆盖率 ≥ 85%）
- [ ] 必须通过静态代码分析（SonarQube）
- [ ] 核心逻辑必须有人工 Code Review 记录
- [ ] 边界条件和异常处理必须人工审查

#### 4.2 智能客服质量指标

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 回答准确率 | ≥ 85% | 用户点赞率 + 人工抽检 |
| 问题解决率 | ≥ 70% | 无需转人工即可解决的对话占比 |
| 转人工率 | ≤ 30% | 转人工会话数 / 总会话数 |
| 平均响应时间 | < 3 秒 | 从用户发送到 AI 开始流式返回的时间 |
| 用户满意度 | ≥ 4.0/5.0 | 对话后评分 |
| Token 成本/会话 | ≤ ¥0.05 | 总 token 费用 / 总会话数 |
| AI 幻觉率 | ≤ 5% | 人工抽检发现"瞎编"的比例 |

#### 4.3 转人工触发条件（组合策略）

| 触发条件 | 优先级 | 说明 |
|----------|--------|------|
| 用户主动要求 | P0 | 用户明确说"转人工"、"找客服"等 |
| AI 连续 2 次答不出 | P1 | AI 返回"我不知道"或置信度<0.5 |
| 用户情绪负面 | P1 | 识别到用户愤怒、失望等情绪 |
| 敏感话题 | P1 | 投诉、退款、法律、安全等 |
| VIP 用户 | P2 | 识别到是高价值用户 |
| 复杂业务场景 | P2 | 涉及多个系统、需要跨部门协调 |

## 🎓 关键洞察

### 技术视角（允灿）
- AI 拉平了"知识差"，但放大了"思维差"
- 护城河从"我知道什么"变成"我能用 AI 解决什么问题"
- 建议：减少死记硬背，增加架构设计和 Prompt 工程能力

### 设计视角（少平）
- AI 交互设计应遵循"透明、可控、渐进"原则
- 明确标识 AI 身份，展示 AI 的"思考过程"
- 设计"人机边界"，任何时候都要给用户"出路"

### 质量视角（少锋）
- QA 角色从"找 Bug 的人"变成"质量策略设计者"
- AI 时代测试金字塔需要调整（探索性测试价值上升）
- AI 应用要在设计阶段考虑可测试性

### 产品视角（易达）
- AI 功能优先级：智能客服 P0，智能搜索 P0
- 转人工触发条件：组合策略（用户主动、AI 答不出、情绪识别等）
- 产品策略：短期单点突破 → 中期能力复用 → 长期 AI 原生

### 管理视角（美娜）
- 采用"业务交付 + 能力建设"双轨并行策略（70% + 30%）
- 转型期分三个阶段：探索期（Q2）、融合期（Q3）、规模化期（Q4）
- 建立 AI 协作知识库，鼓励知识沉淀和分享
