## 📐 技术标准与规范

### 1. API 规范（智能客服项目）

**流式响应格式（SSE）**：
```json
data: {"type": "start", "conversationId": "abc123"}
data: {"type": "thinking", "stage": "searching_knowledge", "message": "正在检索知识库..."}
data: {"type": "chunk", "content": "你好，", "contentType": "text", "tokenUsage": 10}
data: {"type": "chunk", "content": "https://example.com", "contentType": "link"}
data: {"type": "end", "totalTokens": 50, "latency": 1200}
data: {"type": "error", "code": "RATE_LIMITED", "message": "API 限流，请稍后重试"}
```

**错误码定义**：
| 错误码 | HTTP 状态码 | 说明 | 前端处理建议 |
|--------|------------|------|-------------|
| `LLM_TIMEOUT` | 504 | LLM API 超时 | 显示"AI 正在思考，请稍后" |
| `RATE_LIMITED` | 429 | 触发限流 | 显示"请求太多，稍后再试" |
| `INVALID_PROMPT` | 400 | Prompt 违规 | 显示"问题不合规，请重新提问" |
| `CONTEXT_TOO_LONG` | 413 | 上下文超出限制 | 提示"对话太长，建议开启新话题" |
| `SERVICE_UNAVAILABLE` | 503 | AI 服务不可用 | 降级到人工客服或普通搜索 |

**置信度设计**：
```json
{
  "confidence": 0.85,
  "confidenceLevel": "HIGH",
  "sources": ["doc_123", "doc_456"],
  "suggestions": ["建议联系销售确认最新报价"]
}
```
- `confidence >= 0.8`：正常显示
- `0.5 <= confidence < 0.8`：显示"仅供参考"提示
- `confidence < 0.5`：显示"建议转人工确认"按钮

### 2. AI 代码审查清单

**代码质量**：
- [ ] 是否有清晰的注释说明 AI 生成的部分
- [ ] 是否通过了静态代码分析（SonarQube）
- [ ] 单元测试覆盖率是否达标（建议>85%）
- [ ] 是否有边界条件处理（AI 容易忽略）

**逻辑正确性**：
- [ ] 核心业务逻辑是否人工审查过
- [ ] 异常处理是否完整（AI 经常只写 happy path）
- [ ] 是否有安全漏洞（SQL 注入、XSS 等）

**可维护性**：
- [ ] 代码风格是否符合团队规范
- [ ] 是否有过度设计或冗余代码
- [ ] 是否便于后续人工维护

### 3. AI 项目质量门禁

**设计阶段门禁**：
- [ ] 测试人员参与需求评审
- [ ] 测试人员参与交互设计评审
- [ ] 定义清晰的验收标准（含 AI 特有指标）
- [ ] 确定"转人工"的触发条件和流程

**开发阶段门禁**：
- [ ] AI 代码有明确标注（`// Generated by AI, reviewed by XXX`）
- [ ] 单元测试覆盖率 ≥ 85%
- [ ] 完成 Mock LLM API 的测试环境搭建
- [ ] 录制至少 10 个典型对话场景用于回归测试

**上线阶段门禁**：
- [ ] 通过所有自动化测试用例
- [ ] 通过探索性测试（至少 2 轮）
- [ ] 完成性能测试（并发 ≥ 100，响应时间 < 3 秒）
- [ ] 完成安全测试（Prompt 注入、敏感内容过滤）
- [ ] 灰度发布计划已审批（先 10% 用户，观察 1 周）

### 4. AI 特有质量指标

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| **回答准确率** | ≥ 85% | 用户点赞率 + 人工抽检 |
| **问题解决率** | ≥ 70% | 无需转人工即可解决的对话占比 |
| **转人工率** | ≤ 30% | 转人工会话数 / 总会话数 |
| **平均响应时间** | < 3 秒 | 从用户发送到 AI 开始流式返回的时间 |
| **用户满意度** | ≥ 4.0/5.0 | 对话后评分 |
| **Token 成本/会话** | ≤ ¥0.05 | 总 token 费用 / 总会话数 |
| **AI 幻觉率** | ≤ 5% | 人工抽检发现"瞎编"的比例 |

### 交互设计原则

**透明性**：
- 明确标识 AI 身份（"🤖 智能助手（AI）"）
- 展示 AI 的"思考过程"（检索中、生成中）
- 承认不确定性（低置信度时提示"建议人工确认"）

**可控性**：
- 随时中断 AI 回复
- 快速转人工客服
- 清除对话历史重新开始
- 对 AI 回复点赞/点踩（反馈机制）

**渐进性**：
- 首次使用有简单的引导（示例问题）
- 根据用户行为逐步展示高级功能
- 错误时提供降级方案

### "三级响应"机制

| 级别 | 场景 | 处理方式 |
|------|------|----------|
| **Level 1**（80%） | 常见问题、产品信息、简单操作 | AI 直接回答 |
| **Level 2**（15%） | 敏感信息、低置信度、复杂业务 | AI 回答 + 人工确认提示 |
| **Level 3**（5%） | 用户要求转人工、AI 答不出、负面情绪、投诉退款 | 直接转人工 |

## 💡 创新观点与洞察

### 1. 技术护城河的演变（允灿）

**过去**：护城河 = "我知道你不知道的技术细节"

**现在**：护城河 = "我能用 AI 解决你解决不了的问题"

**结论**：AI 拉平了"知识差"，但放大了"思维差"。学习重点应从死记硬背转向架构设计、业务抽象、Prompt 工程。

### 2. 透明性设计原则（少平）

AI 交互设计应遵循"透明、可控、渐进"三原则：
- **透明**：明确标识 AI 身份，展示思考过程，承认不确定性
- **可控**：用户可随时中断、转人工、反馈
- **渐进**：首次引导、逐步展示、错误降级

### 3. 可测试性设计（少锋）

AI 应用要在设计阶段考虑可测试性：
- **可观测**：记录所有 AI 调用的输入输出
- **可 Mock**：LLM 调用抽象为接口，支持依赖注入
- **可回放**：录制典型对话，用于回归测试
- **可追溯**：每个回答记录来源文档
- **可度量**：记录响应时间、token 消耗、置信度

### 4. 双轨制转型策略（美娜）

**轨道 A**：业务交付（70% 精力）
- 按现有方式交付产品功能
- 适度使用 AI 提效（不强制）
- 保证业务连续性和稳定性

**轨道 B**：能力建设（30% 精力）
- AI 技术学习和实践
- 工具链建设和试点项目
- 知识沉淀和团队分享

### 5. 测试金字塔的演变（少锋）

```
AI 时代测试金字塔：
        /\
       /  \      ← 探索性测试（人工 + AI 辅助）
      /    \
     /______\
    /        \    ← 契约测试（前后端、AI API）
   /__________\
  /            \   ← AI 生成单元测试（大量）
 /______________\
```
